    三大体系

    • 对称多处理器结构(SMP:symmetric Muti-Processor)

    • 非一致存储访问结构(NUMA:Non-Uniform Memory Access)

    • 海量并行处理结构(MPP:Massive Parallel Processing)

    MPP

    大规模并行处理结构。MPP 系统由许多 松耦合的处理单元组成的(处理单元，非处理器)。每个处理单元内的CPU都有自己私有的资源，如总线、内存、硬盘等，且都有操作系统和管理数据库的实例复本。特点在于不共享资源(share-nothing)。

    1.MPP架构特点

    • Share Nothing、节点之间数据不共享，只有通过网络连接实现的协同
    • 每个节点有独立的存储和内存
    • 数据根据某种规则散布到各个节点
    • 计算任务也是会发布到各个节点并行执行，最后再将结果聚合到整体返回
    • 用户使用时会看做整体
    • MPP数据库往往优先考虑 A 一致性，然后是 A 可用性，租后考虑 P 分区容忍
    • MPP架构目前被并行数据库广泛采用，一般通过scan、sort和merge等操作符实时返回查询结果

    2.MPP架构劣势

    • 很难高可用 -> 影响可用性和可靠性

    因为数据按某种规则散布到了各个节点上

    • 节点数=任务并行数 -> 影响扩展性

    一个作业提交时，每个节点都要执行相同任务。而不是向hadoop的MapReduce那样做了根据实际开销进行任务拆分后散发到有资源的几个节点上。

    • 每个客户端同时连接所有节点通信 -> 影响网络

    MPP架构每个节点独立，所以客户端往往需要连接所有节点进行通信。

    • 分区容错性差

    MPP主要考虑CA，最后才是 P。一旦扩展节点太多后，原数据管理十分困难。

    3.MPP 适用场景

    集群规模 100 以内，并发小

    MPP架构目前被并行数据库广泛采用，一般通过scan、sort、和merge等操作符实时返回查询结果

    SMP

    ​ SMP最大的特点是CPU共享所有资源，如内存、IO等。操作系统只有一个，他管理着一个队队列，每个处理器依次处理队列中的进程。如果两个处理器同时请求访问一个资源，由硬件、软件的锁机制去解决资源争用问题。

    ​ 所谓对称多处理器是指服务器中多个 CPU 对称工作，无主次或从属关系。各 CPU 共享相同的物理内存，每个 CPU 访问内存中的任何地址所需时间是相同的，因此 SMP 也被称为一致存储器访问结构(UMP：Uniform Memory Access)。对 SMP 服务器进行扩展的方式包括增加内存、使用更快的 CPU、增加 CPU、扩充I/OI（槽口数与总线数）以及添加更多的外部设备（通常是磁盘存储）。

    SMP的问题

    ​ SMP 服务器的主要特征是共享，系统中所有资源（CPU、内存、I/O等）都是共享的。也正式由于这种特征，导致 SMP 服务器的主要问题，那就是他的扩展能力非常有限。

    ​ 对于 SMP 服务器而言，每一个共享的环节都可能造成 SMP 服务器扩展时的瓶颈，而最受限制的则是内存。由于每个 CPU 必须通过相同的内存总线访问相同的内存资源，因此随着 CPU 数量的增加，内存访问冲突将迅速增加，最终会造成 CPU 资源的浪费，使 CPU 性能的有效性大大降低。（SMP 服务器 CPU利用率最好的情况是 2 至 4 个 CPU）

    适用场景

    当节点之间通信交互较少时，MPP 更适应。集中式数据库 SMP 更适用。

    MPP架构服务器就是由多个 SMP服务器通过若干节点组成，通过网络协同工作。

    NUMA

    NUMA 服务器的基本特征：

    • 具有多个 CPU模块
    • 每个 CPU模块又是由多个 CPU组成，并且具有独立的本地内存、I/O 等
    • 节点之间通过互联模块进行通信交互，每个 CPU可以访问整个系统的内存（NUMA 与 MPP 的重要差别，因为MPP中CPU不可访问其他节点内存）

    显然，访问本地内存的速度将远远高于访问其他节点上的内存的速度，这也是非一致存储访问 NUMA 的由来。由于这个特点，为了更好发挥系统性能，开发应用程序时需要尽量减少不同 CPU 模块之间的信息交互。

    利用 NUMA 技术，可以较好的解决原来 SMP 系统的扩展问题，字啊一个物理节点内可以支持 上百个 CPU。

    NUMA的问题

    由于访问其他节点上的内存的延时远远超过本地内存，因此当 CPU 数量增加时，系统性能无法线性增加。

    NUMA例子

    比较典型的 NUMA 服务器的例子包括 HP 的 SUperdome、SUN15k、IBMp690等。

    Hadoop与MPP的区别

    1.底层数据库

    MPP跑的是SQL，而Hadoop底层处理是MapReduce程序。

    2.扩展程度

    MPP的扩展程度一般是扩展到 100 左右，而 Hadoop 一般可以扩展 1000+。

    CAP理论上的差异：

    在理论计算机科学中，CAP定理（CAP theorem），又称为布鲁尔定理（Brewer's theorem），他指出对于一个分布式计算系统来说，不可能同时满足以下三点：

    • 一致性(Consistency)：

    同一个数据在集群中的所有节点，同一时刻是否都是同样的值。

    • 可用性(Availability)：

    集群中一部分节点故障后，集群整体是否还能处理客户端的更新请求。

    • 分区容忍性(Partition tolerance)：

    是否允许数据的分区，分区的意思是指是否允许集群中的节点之间无法通信。

    因为MPP始终还是DB，一定要考虑到C(consistency)，其次考虑A(Availability)，最后才在可能的情况下尽量做好P(Partition tolerance)。而 Hadoop就是为了并行处理和存储设计的，所以数据都是以文件存储，所以优先考虑的是P，然后是A，最后再考虑C。

    以下几个方面制约了MPP数据库的扩展

    • 1.高可用：

    MPP DB是通过Hash计算来确定数据行所在的物理机器（而Hadoop不需要此操作），对存储位置的不透明导致MPP的高可用很难办。

    • 2.并行任务：

    数据是按照Hash来切分的，但是任务没有。每个任务，无论大小都要到每个节点上走一圈。

    • 文件系统：

    数据切分了，但是文件数没有变少，每个表在节点上一定有一个到多个文件。同时节点数越多，存储的表越多，导致每个文件系统由上万甚至十万个文件。

    • 网络瓶颈：

    MPP强调对等的网络，点对点的连接也消耗了大量的网络宽带，限制了网络上的线性扩展。（比如一台机器要给1000台机器发送信息）。等多的节点并没有提供更高的带宽，反而导致每个组节点间平均宽带降低。

    • 其他关系数据库的枷锁：

    比如锁，日志，权限，管理节点瓶颈等均限制了MPP规模的扩大。
